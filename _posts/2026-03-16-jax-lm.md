## Introduction

This blog aims to provide an end-to-end guide for understanding and implementing sharding, parallelism, and distributed training methods for language modeling using JAX.

We draw heavily upon 2 great resources. The first is the Stanford's [CS336 course](https://cs336.stanford.edu/), Language Modeling from Scratch, which teaches us how to implement language models from scratch in PyTorch. This blog draws heavily from the first homework assignment from CS336. The second resource is the [How to Scale Your Model](https://jax-ml.github.io/scaling-book/) textbook, which explains the theory behind different parallelism methods and distributed training.

The goal of this blog is to bridge the gap between these two resources by:
1. Providing a CS336-like interface,  complete with test cases, to implement a language model in JAX instead of PyTorch
2. Implementing the distributed strategies described in the How to Scale Your Model book in both PyTorch and JAX
3. Providing simple and easily configurable code, allowing us to empirically validate the theoretical compute laws presented in the How to Scale Your Model book.

 This blog assumes that you are at least somewhat familiar with PyTorch and language modeling, as this will allow us to focus more on the things that JAX does differently.

This blog is split into 4 parts. First, we will provide a brief introduction to JAX and NNX. Next, we will implement the basic components of a language model in JAX, while providing comparisons to PyTorch. Then, we will introduce various distributed training methods (DP, TP, FSDP, FSDP+TP) and show how to implement them in JAX. Finally, we will use our code to empirically validate theoretical compute rooflines.

With that being said, let's dive in!

## Intro to JAX and NNX

### A Brief Introduction
JAX is an open-source Python library for high-performance numerical computing developed by Google. It combines a NumPy-like syntax with the ability to run on accelerators (GPUs/TPUs), and also provides an automatic differentiation system for efficiently computing gradients. While JAX was initially developed to be compatible with Google's Tensor Processing Units (TPUs), it is also compatible with GPUs and CPUs and can provide significant speedups on these devices.

So what makes JAX unique?

1. Firstly, JAX uses a **functional programming** model. Unlike PyTorch's stateful, object-oriented approach, JAX adopts pure functions and immutable arrays. As an example, there is no model object that holds parameters. Instead, the model is represented as a nested collection of arrays (a PyTree) and passed explicitly into functions. This makes the execution model more transparent and composable.

2. Secondly, JAX provides a set of very powerful **composable function transformations**. The core idea is that we can write a plain Python or NumPy function and transform it into a more powerful function. The four key transforms are as follows:
    1. `jax.jit`, which compiles a function via XLA for fast execution
    2. `jax.grad`, which differentiates a function
    3. `jax.vmap`, which vectorizes a function over some batch dimension automatically
    4. `jax.pmap`, which is similar to `jax.vmap` but parallelizes a function across devices
    These functions can be composed arbitrarily (such as `jit(vmap(grad(f)))`) and can often provide huge speedups in practice.

3. Lastly, JAX uses the **XLA Compiler** as its backend, which optimizes across the entire computation graph instead of operator by operator as PyTorch does. This allows XLA to perform aggressive fusion and optimizations that span multiple operations and even devices, which can unlock large performance gains.

Let's now take a look at some of the code behind these function transformations!

### `jax.grad`

At the heart of any deep learning library is automatic differentiation (autodiff), and JAX's approach is built around `jax.grad`. `jax.grad` is a transform that takes in a function `f` and returns a new function that returns its gradient. This is different than how PyTorch implements autodiff.

In PyTorch, the computation graph is built incrementally as we perform each operation in the forward pass. To get the backwards pass computation, the PyTorch walks through the graph in reverse.

{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260318102536.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260318102536.png" width="416" zoomable=true %}

Image taken from [PyTorch's Overview of the Autograd Engine](https://pytorch.org/blog/overview-of-pytorch-autograd-engine/)

On the other hand, JAX implements the backwards pass by first examining the trace of the entire program, and then transforms the backwards pass in one go into a reusable function. This allows JAX to optimize across the entire computation graph, which is what enables some of the significant JAX speedups. We will discuss this process in more detail in the next section about `jax.jit`.

**Code**
To see this in code form, let's look at the following example of a simple loss function:
```py
import jax
import jax.numpy as jnp

def loss(params, x, y):
    pred = params @ x
    return jnp.mean((pred - y) ** 2)

grad_fn = jax.grad(loss, argnums=(0,1))
grads = grad_fn(params, x, y)
```
We can define our `loss` function normally, using the NumPy-like operation `jnp.mean` to take the mean of our vector. Then, we can call `jax.grad` on our `loss` function to *transform* into a new function `grad_fn` that computes the gradients of its inputs.

One thing to note is that the new function `grad_fn` will be a wrapper around the original `loss` function. When we call `grad_fn` , JAX will run the forward pass (`loss`) and backwards pass (getting the gradients) in the same call. As a result, `grad_fn` must have the same function signature as the original `loss function`.

To specify *which* arguments to differentiate w.r.t., we can pass in `argnums`. Here, `argnums=(0,1)` corresponds to `params` and `x`, so `grad_fn` returns a tuple of 2 gradients, $\frac{\partial \text{Loss}}{\partial \text{Params}}, \frac{\partial \text{Loss}}{\partial X}$.

If we look carefully, the last two lines can actually be combined into the following form^[We drop the `argnums` here for simplicity, it defaults to 0 which corresponds to `params`.]:
{% highlight python %}
grads = jax.grad(loss)(params, x, y)
{% endhighlight %}
where `jax.grad` acts as a higher order function. If you are familiar with Python decorators, we see that this is actually the exact pattern of a decorator! So an equivalent way of writing the entire code would just be:
{% highlight python %}
import jax
import jax.numpy as jnp

@jax.grad
def loss(params, x, y):
    pred = params @ x
    return jnp.mean((pred - y) ** 2)
{% endhighlight %}
Since all JAX transforms are higher-order functions of the same form, they can all be used as decorators in this way.

A small note: since we often want the loss value in addition to the gradients during training, we typically use `jax.value_and_grad` instead of `jax.grad`. The usage is identical, except the returned function now returns a `(loss, grads)` tuple instead of just `grads`.

### `jax.jit`

`jax.jit` is a transform that *compiles* a function using the XLA compiler. JAX compiles the function the first time it is called and reuses the compiled version for future calls^[The JIT in `jax.jit` stands for **J**ust-**I**n-**T**ime compilation, which refers to the fact that functions are compiled at runtime the first time they are called. This is in contrast to Ahead-Of-Time compilation, where everything is compiled before the program runs.].

This is powerful because the XLA compiler can optimize across the entire function and do things like fusing operations, optimizing the memory layout, and parallelization, in a way that is not possible when we execute the function line by line.

So what actually happens when we compile a function? On the first call, JAX *traces* the function by running it with abstract placeholders. You can think of these as something similar to types: instead of passing in `x = jnp.array([1.0, 2.0, 3.0])`, JAX traces with something like `x = float32[3]`. The idea is that XLA doesn't need to know exact values of arrays to build the computation graph for optimization, it only needs to know the shapes and types of data.

After tracing the function to construct a computation graph, JAX compiles it to optimized XLA bytecode. On subsequent calls, we can just run the compiled binaries instead of the Python code.

{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260307144539.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260307144539.png" width="748" zoomable=true %}

In practice, this can speed up code by a lot. The speedup usually gets larger the more operations we fuse in one `jax.jit` call, which is why many implementations wrap the entire training step (forward + backward + optimizer update) into a single `jax.jit` block.

{% details Additional Notes on Compilation %}

Why do we want to compile? Python can be slow for numerical computation since each operation needs to go through the Python interpreter, which has significant overhead. For example, performing `jnp.sum(x ** 2)` actually requires 2 operations (`**` and `sum`), which are dispatched separately to the GPU/TPU.

Thus, each operation launches its own kernel and must read/write intermediate results from memory, which are both quite expensive. Compilation solves this by seeing the entire computation graph at once, allowing us to merge multiple operations into a single GPU kernel, reorder operations to maximize cache usage, and precompute anything that doesn't depend on inputs (and store it for future calls).

{% enddetails %}

{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260318113944.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260318113944.png" width="900" zoomable=true %}

**Code**
Here is what `jax.jit` might look like in code form:
{% highlight python %}
@jax.jit
def f(x):
    return jnp.sum(x ** 2)

x = jnp.ones((3,))
output = f(x)
{% endhighlight %}

A more realistic training step might look something like this:
{% highlight python %}
@jax.jit
def train_step(params, x, y):
    loss, grads = jax.value_and_grad(loss_fn)(params, x, y)
    params = params - lr * grads
    return params, loss

for batch_x, batch_y in data:
    params, loss = train_step(params, batch_x, batch_y)
{% endhighlight %}
Note that we do the entire forwards, backwards, and parameter updates in a single function compiled by `jax.jit`!

**Common Pitfalls**
However, there are some common pitfalls that we need to be wary of.

Firstly, JAX recompiles functions if the shapes of the input change. Take a look at the following example:
{% highlight python %}
@jax.jit
def f(x):
    return jnp.sum(x ** 2)

f(jnp.ones((3,)))  # compiles for float32[3]
f(jnp.zeros((3,))  # uses cached compiled function since same shape
f(jnp.ones((5,)))  # !! recompiles for float32[5] !!
f(jnp.ones((7,)))  # !! recompiles for float32[7] !!
f(jnp.zeros((3,))  # uses cached compiled function from the first call
{% endhighlight %}
JAX compilation takes into account the input shapes to best optimize the program, so it saves a separate binary per shape. In our example, JAX compiles `f` for the first time with a `float32[3]` input and can reuse this for the second call. However, we trigger a costly recompilation each time our input shape is different!

Secondly, since JAX only sees abstract placeholders and operations during tracing, it can't handle functions that might have a variable number of operations. This means that it can't handle value-dependent control flows, such as `if` statements that depend on runtime values or input arguments. We also can't use `for` loops that may perform a variable number of operations.
{% highlight python %}
@jax.jit
def f(x, n):
    count = 0
    for _ in range(n):
        count += 1  # !! we might have a variable number of ops depending on n !!
    if x > 0:  # !! x is float32[1] at trace time, not a value yet !!
        return x
    return -x
{% endhighlight %}

Finally, any Python-specific side effects (such as print statements) only happen once during the initial trace. This is because the Python code is only run once during initial tracing and future calls will run the compiled binary instead.

{% details Tip %}

To print in a JIT-compiled function, we can use `jax.debug.print` instead of the default `print`

{% enddetails %}

### `jax.vmap`

The final core JAX transforms are for vectorization and parallelization. `jax.vmap` stands for "vectorized map" and transforms a function written for a single example into one that operates over a batch.

{% highlight python %}
import jax
import jax.numpy as jnp

def f(x):
    return jnp.dot(x, x)

batched_f = jax.vmap(f)
batched_f(jnp.ones((8, 4)))
{% endhighlight %}
Equivalently, we can also have this in decorator form:
{% highlight python %}
@jax.vmap
def f(x):
    return jnp.dot(x, x)

f(jnp.ones((8, 4)))
{% endhighlight %}
{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260318125311.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260318125311.png" width="774" zoomable=true %}

An illustration of the above `jax.vmap`code.

The idea here is that we can define a dot product function `f` that operates on a single vector, but then use `jax.vmap` to make it batched. By default, `jax.vmap` maps over the first dimension, but we can also specify which dimensions we want to batch over by passing in the `in_axes` argument.

### `jax.pmap`

`jax.pmap` stands for parallel map. It's similar to `vmap` but operates over devices (different GPUs/TPUs) by running each slice of the batch on a separate accelerator.

{% highlight python %}
n_devices = jax.device_count()

def loss_fn(params, x):
    return jnp.sum(params * x)

parallel_loss = jax.pmap(loss_fn, in_axes=(None, 0))

xs = jnp.ones((n_devices, 4))  # one slice per device. note that the batch dimension must equal `n_devices`
parallel_loss(params, xs)       # runs shard i on device i
{% endhighlight %}
In this example, we pass in `in_axes=(None, 0)`. This means that we want the first argument `params` to replicated across all devices (`None`), and the second argument to be split across the device axis `0`. Since we don't specify a device mesh, our devices form a single `(n_device,)` mesh, so this effectively sends row `i` to device `i`.
{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260318131840.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260318131840.png" width="1020" zoomable=true %}

{% details PyTorch comparison for `jax.pmap` %}

The equivalent code in PyTorch would something like the following:
{% highlight python %}
import torch
import torch.distributed as dist

def loss_fn(params, x):
    return torch.sum(params * x)

def main():
    dist.init_process_group(backend='nccl')
    rank = dist.get_rank()
    world_size = dist.get_world_size()

    params = torch.ones(4).cuda(rank)
    xs = torch.ones((world_size, 4)).cuda(rank)  # full batch, same shape as JAX

    x_local = xs[rank]          # manually shard: each process takes its slice
    loss = loss_fn(params, x_local)
    print(f"rank {rank}, loss: {loss}")

if __name__ == "__main__":
    main()
{% endhighlight %}
and we would need to launch it with `torchrun --nproc_per_node=4 script.py`. As you can see, we would need to manually send shards to their assigned devices and each process would also run its own Python interpreter. Meanwhile, JAX allows us to run all of this in a single process with `pmap` and provides cleaner scalability for more complex distributed methods.

{% enddetails %}

## NNX
Now that we've talked about some of the core concepts behind JAX, let's talk a bit about NNX, which is what we will be using for the rest of this blog.

NNX is a neural network API for JAX. If you are familiar with PyTorch, you can think of NNX as the JAX equivalent of `torch.nn`. Both serve as the standard neural network library for their respective frameworks.

To provide a bit of history, NNX was released in 2024 as an evolution of the original Flax Linen API. A core constraint of core JAX is that everything was functional, which meant that model states needed to be passed explicitly in and out of functions. For larger networks with a lot of moving parts, this can become quite painful and error prone. NNX solves this by automatically handling the mutable states for us, allowing us to write object-oriented code in a PyTorch-like style while still being compatible with JAX transforms.


To learn a bit more about why you should use NNX, you can refer to the official explanation [here](https://flax.readthedocs.io/en/stable/why.html).

For most practical purposes though, most transforms in JAX can be used in NNX by replace `jax` with `nnx`. For example, `@jax.jit` becomes `@nnx.jit`, and `jax.value_and_grad` becomes `nnx.value_and_grad`.

Now that we've discussed the ideas behind JAX and NNX, let's take a look at the code!

## Implementing Our Model

In this section, we will implement a basic version of our Transformer language model in JAX and provide a version of the same code written in PyTorch for comparison. We will begin with a simple version without worrying too much about sharding and distributed training.

Let's start by implementing the simplest component of any modern day neural network: the Linear layer!
### The Linear Layer
The following is how you might implement a linear layer (without biases) in PyTorch:
{% highlight python %}
class Linear(nn.Module):
    def __init__(
        self,
        in_features: int,
        out_features: int,
        device: torch.device | None = None,
        dtype: torch.dtype | None = None
    ):
        super().__init__()
        weights = torch.empty(out_features, in_features, device=device, dtype=dtype)
        std = (2 / (in_features + out_features)) ** 0.5
        nn.init.trunc_normal_(weights, mean=0.0, std=std, a=-3.0 * std, b=3.0 * std)
        self.weights = nn.Parameter(weights)

    def forward(self, x: Tensor) -> Tensor:
        return einsum(x, self.weights, "... d_in, d_out d_in -> ... d_out")
{% endhighlight %}
And here is how you would do it in NNX:
{% highlight python %}
class Linear(nnx.Module):
    def __init__( self,
        rngs: nnx.Rngs,
        in_features: int,
        out_features: int,
        dtype: jnp.dtype = jnp.float32,
    ):
        super().__init__()
        std = (2 / (in_features + out_features)) ** 0.5
        init_fn = nnx.initializers.truncated_normal(stddev=std, lower=-3.0 * std, upper=3.0 * std)
        weights_data = init_fn(rngs.params(), (in_features, out_features), dtype)
        self.weights = nnx.Param(weights_data)

    def __call__(self, x: jnp.ndarray) -> jnp.ndarray:
        return jnp.einsum("...i,io->...o", x, self.weights.get_value())
{% endhighlight %}

{% details Weight Storage Convention %}

In PyTorch, the weights for a Linear layer are typically stored with shape `[Out, In]`, whereas in JAX, the weights are stored as `[In, Out]`.

{% enddetails %}

As you can see, the code looks very similar. Aside from minor differences in syntax and storage convention, there are a few key changes that illustrate some of the philosophical differences between JAX and PyTorch.

1. **Initialization**
In PyTorch, initialization functions usually operate in-place on an already-allocated tensor. This is why we create an empty weights tensor in the desired shape and on the desired device before initializing it. On the other hand, initialization functions in JAX typically return a tensor (`weights_data` in this case), and we need to wrap it explicitly with `nnx.Param()` to make it a parameter object so that JAX knows how to use it for transforms and state management. The actual weight tensor of an `nnx.Param` actually lives in it's `.value` attribute.

This example highlights a core philosophical difference between PyTorch and JAX. PyTorch is designed in an object-oriented style centered around stateful modules, where parameters live inside modules and workflows mutate tensors in place. On the other hand, JAX encourages a more functional style, where parameters are treated as explicit data and common workflows are often represented as functions that produce new values.

2. **Device Placement**
In PyTorch, device placement is explicit. Typically, we create parameters directly on a target device with `device=...` or move them afterwards with `.to(device)`. Meanwhile, in JAX, device placement is implicit and determined automatically by the compiler and any sharding configs we specify.

3. **RNG**
In JAX, RNGs need to be passed in explicitly, and are split/consumed deterministically. On the other hand, RNG is largely global and ambient (i.e. we can set a global RNG seed) and we don't need to pass it explicitly to each function).

In the example above, we use `rngs.params()` in the `init_fn`, which generates an random number from the `params` stream. There are 2 main PRNG key stream names used by Flax NNX’s built-in layers, which are `params` and `dropout`.

**Some other minor differences**:
- PyTorch uses the `forward` convention, which is wrapped by the `__call__` method implemented by the parent class `nn.Module`. JAX typically uses `__call__` directly.

### Multi-head Self Attention
The other building blocks of our model have minor syntax differences compared to PyTorch, but follow the same ideas discussed here. Let's take a look at another example before moving on! This is what code for multi-head self attention (MHA) in PyTorch would look like:
{% highlight python %}
class MultiHeadSelfAttention(nn.Module):
    def __init__(
        self,
        d_model: int,
        num_heads: int,
        rope_theta: float = 1e4,
        max_seq_len: int = 1024,
        device: torch.device | None = None,
        dtype: torch.dtype | None = None,
    ):
        super().__init__()
        assert d_model % num_heads == 0, "d_model must be divisible by num_heads"
        self.num_heads = num_heads
        self.d_k = d_model // num_heads
        self.Q_proj = Linear(d_model, d_model, device=device, dtype=dtype)
        self.K_proj = Linear(d_model, d_model, device=device, dtype=dtype)
        self.V_proj = Linear(d_model, d_model, device=device, dtype=dtype)
        self.O_proj = Linear(d_model, d_model, device=device, dtype=dtype)
        self.register_buffer(
            "causal_mask",
            torch.tril(torch.ones(max_seq_len, max_seq_len, device=device)),
            persistent=False,
        )
        self.rope = RotaryPositionalEmbedding(self.d_k, theta=rope_theta, max_seq_len=max_seq_len, device=device)

    def forward(self, x: Tensor, use_rope: bool = False) -> Tensor:
        batch_size, seq_len, _ = x.shape
        q = self.Q_proj(x).reshape(batch_size, seq_len, self.num_heads, self.d_k).transpose(1, 2)
        k = self.K_proj(x).reshape(batch_size, seq_len, self.num_heads, self.d_k).transpose(1, 2)
        v = self.V_proj(x).reshape(batch_size, seq_len, self.num_heads, self.d_k).transpose(1, 2)

        if use_rope:
            positions = torch.arange(seq_len, device=x.device)
            q = self.rope(q, positions)
            k = self.rope(k, positions)

        attn_out = sdpa(q, k, v, self.causal_mask[:seq_len, :seq_len])
        attn_out = attn_out.transpose(1, 2).contiguous().reshape_as(x)
        return self.O_proj(attn_out)
{% endhighlight %}

And here is what it looks like in JAX!
{% highlight python %}
class MultiHeadSelfAttention(nnx.Module):
    def __init__(
        self,
        rngs: nnx.Rngs,
        d_model: int,
        num_heads: int,
        rope_theta: float = 1e4,
        max_seq_len: int = 1024,
        dtype: jnp.dtype = jnp.float32,
    ):
        super().__init__()
        assert d_model % num_heads == 0, "d_model must be divisible by num_heads"
        self.num_heads = num_heads
        self.d_k = d_model // num_heads
        self.Q_proj = Linear(rngs=rngs, in_features=d_model, out_features=d_model, dtype=dtype)
        self.K_proj = Linear(rngs=rngs, in_features=d_model, out_features=d_model, dtype=dtype)
        self.V_proj = Linear(rngs=rngs, in_features=d_model, out_features=d_model, dtype=dtype)
        self.O_proj = Linear(rngs=rngs, in_features=d_model, out_features=d_model, dtype=dtype)

        self.causal_mask = jnp.tril(jnp.ones((max_seq_len, max_seq_len)))
        self.rope = RotaryPositionalEmbedding(d_k=self.d_k, theta=rope_theta, max_seq_len=max_seq_len)

    def __call__(self, x: Array, use_rope: bool = False):
        batch_size, seq_len, _ = x.shape
        q = self.Q_proj(x).reshape(batch_size, seq_len, self.num_heads, self.d_k).swapaxes(1, 2)
        k = self.K_proj(x).reshape(batch_size, seq_len, self.num_heads, self.d_k).swapaxes(1, 2)
        v = self.V_proj(x).reshape(batch_size, seq_len, self.num_heads, self.d_k).swapaxes(1, 2)

        if use_rope:
            positions = jnp.arange(seq_len)
            q = self.rope(q, positions)
            k = self.rope(k, positions)

        attn_out = sdpa(q, k, v, self.causal_mask[:seq_len, :seq_len])
        attn_out = attn_out.swapaxes(1, 2).reshape(x.shape)
        return self.O_proj(attn_out)
{% endhighlight %}
Aside from the syntax and RNG differences described earlier, the two implementations look basically the same!

However, one interesting difference to note here is how causal masks are handled. In PyTorch, it's common to register the mask as a buffer at initialization:
{% highlight python %}
self.register_buffer(
    "causal_mask",
    torch.tril(torch.ones(max_seq_len, max_seq_len, device=device)),
    persistent=False,
)
{% endhighlight %}
Unlike parameters, buffers are not learnable and not tracked by the optimizer. However, they are still included in the model's state dict and automatically moved with the model during device placement.

In JAX, we can just assign it directly:
{% highlight python %}
self.causal_mask = jnp.tril(jnp.ones((max_seq_len, max_seq_len)))
{% endhighlight %}
This is because device placement is automatic in JAX, so there's no need for a special registration mechanism to ensure the mask follows the model to the correct device. As a bonus, since `jax.jit` treats any closed-over array as a constant in the compiled graph, the mask is never recomputed across calls.

## Implementing the Training Loop

Now that we have our model implemented in JAX, let's take a look at how we can train it!

There are a few major differences in the JAX training loop compared to PyTorch. We'll discuss each one and what it might reveal about the JAX paradigm.

### The Training Step
In PyTorch, our train step looks something like this:
{% highlight python %}
logits = model(inputs)
B, S, V = logits.shape
loss = model_module.cross_entropy_loss(logits.reshape(B * S, V), targets.reshape(B * S))
optimizer.zero_grad()
loss.backward()
optimizer.step()
{% endhighlight %}
Meanwhile, in JAX, it looks something like:
{% highlight python %}
loss, grad_state = train_step(model, optimizer, inputs, targets)
{% endhighlight %}
Where the `train_step` function looks something like:
{% highlight python %}
@nnx.jit
def train_step(model: nnx.Module, optimizer: nnx.Optimizer, inputs: Array, targets: Array):
    def loss_fn(model):
        logits = model(inputs)
        B, S, V = logits.shape
        return cross_entropy_loss(logits.reshape(B * S, V), targets.reshape(B * S))

    loss, grads = nnx.value_and_grad(loss_fn)(model)

    optimizer.update(model, grads)
    return loss, grads
{% endhighlight %}

**Train Step Function**
In PyTorch, we don't need to wrap our training step into a function. Most of our operations operate in place. However, we need to create a `train_step` function in JAX so that we can wrap it with `nnx.jit`.

**Gradient Handling**
In PyTorch, gradients are implicit, since `loss.backward()` accumulates gradients into each parameter’s `.grad` field, and `optimizer.step()` reads those `.grad` buffers and updates parameters in place. As a result, our gradient states are effectively distributed across `{p.grad for p in model.parameters()}`, instead of being a single gradient object `grad` like in JAX. Gradient clipping in PyTorch follows the same philosophy by scaling the `.grad` buffers in place.

On the other hand, gradients are treated as explicit values in JAX. To get our `grad` object, we use `nnx.value_and_grad`, which is a [transformation](#intro-to-jax-and-nnx) that turns the `loss_fn` into a function that also returns gradients. Since JAX transforms act as higher order functions, we need to define a loss function and pass it in as input to `nnx.value_and_grad` instead of just calling `loss.backward()` like we do in PyTorch.

###  Gradient Clipping and LR Schedule
If we wanted to add gradient clipping and a learning rate schedule in PyTorch, they would be called separately from our optimizer. In JAX, they would be processed by the optimizer internally.

**PyTorch**
Here's what it would look like in PyTorch:
{% highlight python %}
def build_lr_scheduler(
    optimizer: torch.optim.Optimizer,
    min_learning_rate: float,
    warmup_iters: int,
    cosine_annealing_iters: int,
    warmup_start_factor: float = 0.1,
) -> torch.optim.lr_scheduler.LRScheduler:
    cosine_decay_iters = cosine_annealing_iters - warmup_iters
    warmup_scheduler = torch.optim.lr_scheduler.LinearLR(
        optimizer,
        start_factor=warmup_start_factor,
        end_factor=1.0,
        total_iters=warmup_iters,
    )
    cosine_scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(
        optimizer,
        T_max=cosine_decay_iters,
        eta_min=min_learning_rate,
    )
    scheduler: torch.optim.lr_scheduler.LRScheduler = torch.optim.lr_scheduler.SequentialLR(
        optimizer,
        schedulers=[warmup_scheduler, cosine_scheduler],
        milestones=[warmup_iters],
    )
    return scheduler
{% endhighlight %}
and the new `train_step` would look something like this:
{% highlight python %}
logits = model(inputs)
B, S, V = logits.shape
loss = model_module.cross_entropy_loss(logits.reshape(B * S, V), targets.reshape(B * S))
optimizer.zero_grad()
loss.backward()
if gradient_clip > 0:  # gradient clipping
    torch.nn.utils.clip_grad_norm_(model.parameters(), float(gradient_clip))
optimizer.step()
if scheduler is not None:  # scheduler handling
    scheduler.step()
{% endhighlight %}
In the function `build_lr_scheduler`, we create a scheduler object that we chains together a linear warmup schedule with a cosine annealing schedule. To use this, we call `scheduler.step()`, which computes new LR values and mutates the optimizer's `param_groups`' `lr` attributes in place.

**JAX**
In contrast, gradient clipping and the learning rate scheduler are used as inputs during optimizer construction in JAX. Our optimizer construction would look something like:
{% highlight python %}
lr_schedule = optax.warmup_cosine_decay_schedule(
    init_value=0.0,
    peak_value=a_max,
    warmup_steps=warmup_iters,
    decay_steps=decay_iters,
    end_value=a_min
)

def build_optimizer_transform(
    optimizer_config: dict,
    gradient_clip: float,
    lr_schedule: optax.Schedule,
) -> optax.GradientTransformation:
    transforms = []
    if gradient_clip > 0:
        transforms.append(optax.clip_by_global_norm(float(gradient_clip)))
    transforms.append(
        adamw(
            lr=lr_schedule,
            betas=tuple(optimizer_config["betas"]),
            weight_decay=float(optimizer_config["weight_decay"]),
            eps=float(optimizer_config["eps"]),
        )
    )
    return optax.chain(*transforms)

optimizer = nnx.Optimizer(
    model,
    build_optimizer_transform(
        optimizer_config,
        gradient_clip=gradient_clip,
        lr_schedule=lr_schedule,
    ),
    wrt=nnx.Param,
)
{% endhighlight %}
In JAX, the optimizer is treated as a series of Optax transformations. We can first construct our pipeline by adding gradient clipping followed by AdamW. Note that we pass in our learning rate schedule `lr_schedule` as an argument to AdamW during initialization. Then, we can create an `nnx.Optimizer` using our `model`, the pipeline we just created, and the `wrt` argument that specifies what to optimize. Finally, we can call this without changing our code by using `optimizer.update(model, grads)`, which will handle the gradient clipping and LR updates internally. Note that unlike PyTorch, we don't need to call `scheduler.step()` or `clip_grad_norm_()` explicitly in each step.

## Intro to Distributed Training

So far, we have assumed that both the model and data batch fit on a single device. However, once the model state or activations become too large, or once we want more throughput than a single device can provide, we need to spread the work across multiple GPUs or TPUs. Doing so can give us more total memory and compute, but it also introduces a new bottleneck: communication. As a result, a core problem in distributed training is deciding what should stay local to each device and what needs to be communicated.

Different parallelism strategies answer this question in different ways. The goal of this section is to build some intuition for some common parallelism strategies, and in the next section we will show how to implement them in JAX. As we'll see, JAX makes distributed training easy!

{% details Note %}

This section is not meant as a complete guide to distributed training. We recommend starting with the [How to Scale Your Model](https://jax-ml.github.io/scaling-book/) book for a more complete guide.

{% enddetails %}

### Motivation
Each device has finite memory and finite compute, which may become bottlenecks as we try to scale our model. Distributing work across devices increases the total memory and compute available to us, but there is no free lunch since the added communication can be costly. A good distributed strategy therefore balances two competing goals: doing as much useful local computation as possible, while minimizing how much data must move between devices. In practice, the right choice depends on both the model size and the hardware we are training on.

To keep the notation readable, we will use the following sharding notation: `B_X` means the leading axis is sharded across the device axis `X`, and `D_Y` means the hidden dimension is sharded across the device axis `Y`. For simplicity, we will only show weight gradients, not input gradients.

Before looking at the parallelism schemes, it helps to name the three collectives that appear repeatedly. An **all-gather** reconstructs a full tensor from shards. A **reduce-scatter** sums contributions across devices and leaves each device with only one shard of the result. An **all-reduce** sums across devices and returns the full summed result to every device; equivalently, it can be viewed as a reduce-scatter followed by an all-gather.

With that being said, let's take a look at some of the most common distributed training methods!

### Standard Training
As a starting point, let's take a look at a linear MLP. We can represent this with the following:

$$
\text{In}[B, D] \cdot_{D} W_{1}[D, F] \cdot_{F} W_{2}[F, D] \to \text{Out}[B, D]

$$
On a single device, the full input, weights, activations, gradients, and optimizer state all live together. The forward pass computes `Out`, the backward pass computes `dW`, and the optimizer updates `W`. This is the simplest case, and it gives us a reference point for the distributed strategies below.

{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260320011439.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260320011439.png" width="1063" zoomable=true %}

### Data Parallel (DP)
Data parallelism is the simplest distributed strategy. We shard the input across the batch dimension and keep a full copy of the weights on every device. Each device therefore runs the same forward and backward pass, but on a different slice of the batch:

$$
\text{In}[B_X, D] \cdot_D W_{1}[D, F] \cdot_{F} W_{2}[F,D] \rightarrow \text{Out}[B_X, D]

$$
After the backward pass, each device has a local contribution to the weight gradient from its own slice of the batch. To recover the gradient for the full batch, we can all-reduce those local gradients across devices. Every device then receives the same global gradient, applies the same update, and keeps its weight copy in sync with the others. DP is great because of its simplicity since the forward pass requires no communication.
{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260320011922.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260320011922.png" width="1064" zoomable=true %}

### FSDP
You might notice a disadvantage of DP is that each device must hold a complete copy of the weights, gradients, and optimizer state. FSDP tries to solve this by sharding the model state across devices. Each device only stores permanently its own shard of the weights and optimizer state:

$$
\text{In}[B_X, D] \cdot_D W_{1}[D_{X}, F] \cdot_{F} W_{2}[F,D_{X}] \rightarrow \text{Out}[B_X, D]

$$
However, because our weights are sharded along the contracting dimension, we can't perform the forward pass matmul with only our local weight shard. As a result, we need to all-gather the parameter shards so that each device can materialize the full weight matrix before the layer runs. After this forward pass computation, the temporary full copy can be discarded.

In the FSDP backwards pass, each device will only have the gradients of the local slice of the input batch, similar to DP. However, instead of all-reducing the full gradient like DP, we can reduce-scatter the gradient so that each device keeps only the gradient shard corresponding to the parameters it owns. The key idea is that the FSDP lowers the *persistent* per-device memory footprint, even though each device still temporarily materializes the full layer weights when the layer is executing.
{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260320012100.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260320012100.png" width="1162" zoomable=true %}

In this image, we split both our batch and weights across our devices. In order to perform a forward pass, each device gathers the other needed weights (the crosshatched blue and orange weights) before the forward pass, and can discard them afterwards.
### TP
One important observation is that FSDP mostly reduces persistent parameter memory but not necessarily the peak per-device memory during the forward pass. As a result, if our model or sequence length is so large that our activations don't fit in a single forward pass, FSDP won't help us much. Tensor parallelism (TP) takes a different approach by **sharding the computation itself**, rather than only sharding stored parameters. Instead of moving weights across devices, we shard the weights and **distribute the computation of a single layer across devices**. Here is how we can represent TP:

$$
\text{In}[B, D_{Y}] \cdot_D W_{1}[D, F_{Y}] \cdot_{F} W_{2}[F_{Y},D] \rightarrow \text{Out}[B, D]
$$

{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260320012754.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260320012754.png" width="1146" zoomable=true %}

Here, we shard along the feature dimension (denoted by $Y$), so each device holds only a slice of the intermediate activations and weights.
In this setup, each device computes only its local contribution to the output. Importantly, since each device can compute the gradient for its local weight shard directly, **no communication is required for weight gradients**.

However, communication is still required elsewhere. After the forward pass, each device holds only a partial output from its local weight shards, so we must all-reduce our output across devices to obtain the full output needed for the loss. In the backward pass, the same issue appears for the input gradients: each device computes only a partial contribution to $\nabla X$, so we again need an all-reduce to combine these contributions (although input gradients are not shown in the diagram for simplicity).
### FSDP+TP
Can we take FSDP and TP a step further? Turns out we can actually combine them by assigning each to different device axes! Along the $X$ axis, we use FSDP, while along the $Y$ axis, we use TP:

$$
\text{In}[B_{X}, D_{Y}] \cdot_D W_{1}[D_{X}, F_{Y}] \cdot_{F} W_{2}[F_{Y},D_{X}] \rightarrow \text{Out}[B_{X}, D_{Y}]

$$
This combination is useful because the two strategies help each other. FSDP shards the batch across `X`, which reduces the amount of activation data that TP must move. TP shards the weights across `Y`, which reduces the amount of parameter data that FSDP must move. The tradeoff is that we now pay both activation communication and weight communication, so the implementation is more complex and communication costs are higher overall. For a discussion on how to balance FSDP and TP, and when to use it, we recommend reading the related section in the [How to Scale Your Model](https://jax-ml.github.io/scaling-book/training/#combining-fsdp-and-tensor-parallelism) book.

{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260320015521.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260320015521.png" zoomable=true %}

This image might be a bit hard to read due to the size, but you can find the full-size image [here](#TODO).

In this example, we expand the number of devices we use to 4, since we need to use at least 2 devices to perform FSDP and another 2 to perform TP. On this 2x2 device mesh, each device owns 1 quadrant of the input and 1 quadrant of the weights.
Before the matmul, TP all-gathers the inputs across $Y$ to get the full data dimension $D$. Then, we can use FSDP to gather our weight matrices across the $X$ axis. Then, we perform a TP-like All-Reduce for our output shards across the $Y$ axis, then finally reduce scatter our gradients across $X$ in an FSDP-like way.

In this example, we expand to 4 devices arranged in a 2x2 mesh. We use one axis $X$ for FSDP and the other axis $Y$ for tensor parallelism. Each device therefore holds a shard of both the **input** and the **weights**, corresponding to its position in the mesh.

During the forward pass, we:
1. TP all-gather over $Y$
    Each device initially only holds a slice of the input along the $D$ dimension. To perform the first matmul, we all-gather the input across the $Y$ axis, so every device within an $X$-group has access to the full dimension $D$ required for computation.
2. FSDP all-gather over $X$ (weights)**
    The weights are sharded across the $X$ axis. Before the matmul, we all-gather the weight shards across $X$ so that each device has the full parameters needed locally for its computation.
3. Local matmul + TP partitioned output
    Each device computes its local portion of the output, which remains sharded across $Y$.
4. TP reduce-scatter (or all-reduce) over $Y$ (outputs)
    Since each device only produces a partial contribution to the output, we must combine these contributions across $Y$. In practice, this is typically implemented as a reduce-scatter, so that the output remains sharded while still being correct.

Now that we've learned about some common distributed training paradigms, let's see how we can implement them in JAX!

## Implementing Distributed Training

Let's take a look at how we can implement distributed training in JAX. Compared to PyTorch, it's actually surprisingly easy! Instead of specifying the low-level communications between the devices, we can just specify how we want our model to be split, and JAX will take care of the rest.

In this section, we will go over:
1. The device mesh
2. Sharding annotations
3. How to use sharding annotations to initialize the model
4. How to define our sharding annotations for each parallelism mode

### The Basics
First, let's introduce some important concepts that we'll need to allow our model code to support distributed training.

#### The Device Mesh
In order to distribute our model or data across multiple devices, we first need to know what devices we have and how they are arranged. We can specify how they are arranged by creating a "device mesh", which would allow us to arrange our devices similar to how we would arrange a matrix.
 For example, if we have 8 devices, we can either arrange our devices in a \[1, 8\] shape, a \[4, 2\] shape, a \[2, 4\] shape, and so on. Each dimension of this device mesh is called an "axis", and the conventional approach is to name the X axis the "data" axis and the Y axis the "tensor" axis.
 
{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260316200546.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260316200546.png" width="569" zoomable=true %}

We can define our device mesh as following:
{% highlight python %}
from jax.sharding import Mesh

def create_mesh(mesh_shape: list[int], mesh_axis_names: list[str]) -> Mesh:
    auto_mesh = jax.make_mesh(tuple(mesh_shape), tuple(mesh_axis_names))
    return auto_mesh

mesh = create_mesh((4, 2), ("data", "tensor"))
{% endhighlight %}
Here, we pass in `(4, 2)` as our mesh shape to create a 4x2 device mesh, with the `"data"` being the X device axis and `"tensor"` being the Y device axis.

#### Sharding Annotations
Now that we have our device mesh, our next step is to specify how we want our data and model components to be sharded. These sharding annotations need to be passed in at model initialization so that JAX knows how to distribute tensors across devices.

To do this, we wrap our existing initialization functions with `nnx.with_partitioning(init_fn, sharding)`, which will attach `sharding` annotations to the variable created with `init_fn`.

In practice, here's what it might look like:
{% highlight python %}
class Linear(nnx.Module):
    def __init__(self, rngs: nnx.Rngs, in_features: int, out_features: int,
    sharding: Sharding | None = None, dtype: jnp.dtype = jnp.float32,
    ):
        super().__init__()
        std = (2 / (in_features + out_features)) ** 0.5
        # We use `nnx.with_partitioning` here!
        init_fn = nnx.with_partitioning(
            nnx.initializers.truncated_normal(stddev=std, lower=-3.0 * std, upper=3.0 * std), sharding
        )
        weights_data = init_fn(rngs.params(), (in_features, out_features), dtype)
        self.weights = nnx.Param(weights_data)

    def __call__(self, x: jnp.ndarray) -> jnp.ndarray:
        return jnp.einsum("...i,io->...o", x, self.weights.get_value())
{% endhighlight %}
The only differences from a standard Linear layer are that:
1. We wrap the original `nnx.initializers.truncated_normal()` call with the `nnx.with_partioning()` function
2. We pass in the `sharding` as input

But you might be wondering, what does `sharding` look like? It's actually just a tuple of strings, one per matrix dimension. Each string names the device axis that dimension is distributed over, and `None` means that the dimension is not sharded across the axis, effectively replicating it.

To make this concrete, consider our 4×2 device mesh and a weight matrix $W[D, F]$. A sharding of `(None, None)` replicates both dimensions — the D dimension across the X axis and the F dimension across the Y axis — meaning the full matrix is replicated on every device.

With `("data", None)` instead, the D dimension is sharded across the 4 devices on the X axis while F is replicated along the Y axis. This means the first D/4 rows live on devices 0 and 1, the next D/4 rows on devices 2 and 3, and so on.

The diagram below illustrates several sharding configurations for our 4×2 mesh. Note that $\text{Device A, B, C}$ indicates that slice of the matrix is replicated across devices $A$, $B$, and $C$.

{% include figure.liquid loading="eager" path="assets/img/jax-lm/pasted-image-20260316203617.png" class="img-fluid rounded z-depth-1" alt="Pasted image 20260316203617.png" width="717" zoomable=true %}

#### Initializing our Model
Now that we've annotated each component of our model with `nnx.with_partitioning`, we need to actually initialize our model so that our tensors land on the right devices. The process is roughly as follows:
1. Initialize model and optimizer inside a JIT compiled function
2. Extract the sharding annotations we attached earlier and apply them as constraints
Because the entire function is JIT-compiled, the XLA compiler can see both the initialization and sharding at once to place the tensors on the correct devices from the start without needing multiple copies or additional communication.

Here's what it might look like in code:
{% highlight python %}
    @nnx.jit
    def init_model_and_optimizer(rngs: nnx.Rngs):
        model = TransformerLM(
            rngs=rngs,
            model_config,
            sharding_config=sharding_config,
        )

        optimizer = nnx.Optimizer(
            model,
            build_optimizer_transform(
                optimizer_config,
                ...
            ),
            wrt=nnx.Param,
        )

        # Shard model parameters according to their partition specs
        model_state = nnx.state(model)
        model_shardings = nnx.get_named_sharding(model_state, mesh)
        model_sharded_state = jax.lax.with_sharding_constraint(model_state, model_shardings)
        nnx.update(model, model_sharded_state)

        # Shard Optimizer State (inherits sharding from params)
        optimizer_state = nnx.state(optimizer, nnx.optimizer.OptState)
        optimizer_shardings = nnx.get_named_sharding(optimizer_state, mesh)
        optimizer_sharded_state = jax.lax.with_sharding_constraint(optimizer_state, optimizer_shardings)
        nnx.update(optimizer, optimizer_sharded_state)

        return model, optimizer
{% endhighlight %}
Let's walk through some of the key steps.
1. `nnx.state(model)` extracts the model's state as a PyTree
2. `nnx.get_named_sharding(model_state, mesh)` reads the annotations we attached previously with `nnx_woth_partitioning` and builds a PyTree of sharding specs, one per tensor
3. We pass both PyTrees (parameters and sharding) into `jax.lax.with_sharding_constraint`, which tells the compiler the required layout for each tensor, allowing it to work backwards and optimize placement for the entire initialization
4. Finally, `nnx.update(model, model_sharded_state)` writes the sharded state back into the original model object
The optimizer state follows essentially the same process.

Note that the entire function is wrapped with `@nnx.jit` and would not work without it. This is because the sharding constraints are compiler directives that only take effect during compilation. Without JIT, JAX would not have a compilation step in which to apply the sharding.
#### Data Loading
Beyond the model itself, we also need to shard the input data. The approach depends on the parallelism strategy: in TP, each device needs the full batch (since the model is split, not the data), while in DP, FSDP, and FSDP+TP, we split the batch across the data axis so each device processes a different slice.

We can capture this logic with a simple helper:
{% highlight python %}
def get_batch_sharding_for_mode(mode: str) -> Sharding:
    if mode == "tp":
        return (None, None)       # replicate across all devices
    if mode in ("dp", "fsdp", "fsdp_tp"):
        return ("data", None)     # split along data axis, replicate along tensor axis

batch_partition_spec = P(*model_module.get_batch_sharding_for_mode(sharding_mode))
{% endhighlight %}
Then, at data loading time, we use `jax.device_put` to place the batch according to this spec:
{% highlight python %}
def get_batch_from_memmap(
    rngs: nnx.Rngs,
    dataset: DatasetLike,
    batch_size: int,
    context_length: int,
    mesh: Mesh | None = None,
    batch_partition_spec: P | None = None,
) -> tuple[Array, Array]:
    inputs, targets = model_module.get_batch(
        rngs,
        dataset.data,
        batch_size,
        context_length,
    )

    # Place batch on devices according to the partition spec
    if mesh is not None:
        with jax.set_mesh(mesh):
            inputs = jax.device_put(inputs, batch_partition_spec)
            targets = jax.device_put(targets, batch_partition_spec)
    return inputs, targets
{% endhighlight %}

Rather than assigning every tensor a completely separate sharding rule, we can group tensors by the role they play.

In particular, many weight matrices fall into one of two common patterns:
1. Projection out of the model dimension $D$ into some larger or different dimension. This includes $W_{Q}, W_{K}, W_{V}$ and $W_{1}, W_{3}$, which have shape $[D, *]$. We can refer to this pattern as `dense_in`
2. Projection back into the model dimension $D$ from some intermediate dimension. This includes $W_{O}$ and $W_{2}$. We will refer to this pattern as `dense_out`.
This naming is useful because these matrices often share the same partitioning logic under a given parallelism strategy, even if their exact shapes differ.

{% details What about Embedding and LM Head? %}

You might notice that the Embedding and LM Head (the unembedding matrix) are not grouped into `dense_in` and `dense_out`. This is because they often play a special role because they involve the vocabulary dimension $V$ instead of hidden dimensions like $D$, $F$, or $N*H$. As a result, they often need different sharding rules since the embedding performs lookup over the vocabulary rows, while the LM head produces vocab logits, which may be sharded across vocab and may require special handling for softmax and cross-entropy. Finally, these 2 layers are often also weight-tied in many language models, which adds an extra layout constraint. For these reasons, it's cleaner to keep them as separate sharding categories.

{% enddetails %}

\text{In}[B_X, D] \cdot_D W_{1}[D, F] \cdot_F W_{2}[F, D] \rightarrow \text{Out}[B_X, D]
$$

We want to split our batches across the $X$, or "data" axis, and replicate our weights otherwise.
